# Deploy & hosting runbook

Same flow as the other NEON series apps: a **hosted Shiny app** + a **GitHub-Pages landing
page** (`docs/`) with an og social card and a cold-start pre-warm.

```
  Posit Connect Cloud (or shinylive)          GitHub Pages  (docs/)
  └─ runs global/ui/server + R/ + www/        └─ landing page → "Launch" → app URL
     + data/*.rds indexes + national_onsets      + og:image social card
     + data/sites/*.rds bundles (46 sites)       + fetch() pre-warm ping (cold-start fix)
```

The landing page lives at [`docs/index.html`](docs/index.html) and the social card at
[`docs/og-image.png`](docs/og-image.png) (regenerate with `Rscript scripts/make_og_image.R`).
After you deploy the app, set the app URL in **one place** — the `APP_URL` constant near the
bottom of `docs/index.html` — then re-publish Pages.

---

## 1 — Deploy the app: Posit Connect Cloud (recommended; git-backed, auto-redeploys)

Connect Cloud deploys an R app straight from this public GitHub repo using `manifest.json`
(generated by [`scripts/write_manifest.R`](scripts/write_manifest.R)).

The manifest is **lean and bundle-only**: `global/ui/server.R`, `R/`, `www/`, the precomputed
`data/*.rds` indexes + `data/national_onsets.rds`, the `data/sites/*.rds` bundles, and the demo
sample — **91 CRAN packages, NOT `neonUtilities` / `arrow` / `data.table`.** `neonUtilities` is referenced by a *computed*
name in `global.R` (`.NEON_PKG`) so the dependency scanner never pins it (no wasm build; a live
NEON pull on a cold worker is a hang risk). The deployed app runs entirely from the bundled
`.rds` files; the optional live-fetch toggle only appears where `neonUtilities` is installed
(local dev). Force bundle-only anywhere with `PHE_LIVE=0`.

1. Sign in at <https://connect.posit.cloud> with GitHub.
2. **New Content → Shiny → from Git**, pick `tgilbert14/NEON-Plant-Phenology-Explorer`, branch
   `master`. Connect reads `manifest.json` (`appmode: shiny`) and auto-detects the multi-file
   Shiny app (`global.R` + `ui.R` + `server.R`) — no single "primary file" to choose.
3. It restores packages and serves. **Auto-republishes on every push to `master`.**
4. Copy the published share URL → set `APP_URL` in `docs/index.html` (and the README badge).

Regenerate the manifest only when **runtime dependencies** change (not on a data refresh):
```r
# R with the app's runtime packages:
Rscript scripts/write_manifest.R   # writes manifest.json, prunes + HARD-GATES neonUtilities/arrow/data.table (stop()s if any leaked)
```

**Cold start:** the free tier sleeps. The landing page's pre-warm `fetch(APP_URL)` on load wakes
it while the visitor reads, so it's usually warm by the time they click Launch.

## 2 — Publish the landing page (GitHub Pages)

1. Repo **Settings → Pages → Source: Deploy from a branch → `master` / `/docs`**.
2. Live at `https://tgilbert14.github.io/NEON-Plant-Phenology-Explorer/` (the og:image meta
   already uses this URL). For a custom subdomain, add a `CNAME` file in `docs/` + a DNS CNAME
   with the cloud proxy off for validation.
3. `docs/og-image.png` (1200×630) is regenerated by `scripts/make_og_image.R`.

## 3 — Keep the data fresh (automatic)

[`.github/workflows/refresh-data.yml`](.github/workflows/refresh-data.yml) re-pulls every NEON
terrestrial site (`scripts/fetch_all_phe.R`), rebuilds the bundles + indexes
(`scripts/bundle_phe_data.R`), and commits to `master` — which **auto-redeploys** the Connect
Cloud app. It runs the first Saturday night of each month (off-peak Arizona time) and on demand
(Actions tab → Run workflow). Requires one repo secret: **`NEON_TOKEN`** (Settings → Secrets and
variables → Actions; your token from <https://data.neonscience.org/myaccount>). Without it the
fetch still works but is rate-limited.

## Option B — Shinylive / WebAssembly (best long-term: static, zero server)

The deployed app has **no hard runtime dependency on `neonUtilities`** (read-only bundled `.rds`),
so it's a shinylive candidate — it could serve from GitHub Pages alongside the landing page.
**Gate: every package needs a wasm binary** (check <https://repo.r-wasm.org/> — especially
`leaflet`, `plotly`, `bslib`, `zip`). Also weigh the **bundle size** (46 site `.rds` + indexes
ship to the browser).

```r
# install.packages("shinylive")
shinylive::export(".", "docs/app")   # static site under docs/app/; then APP_URL = "app/"
```

If wasm + size check out, this is the cheapest home and removes the cold start. Otherwise stay on
Connect Cloud.

## Optional — shinyapps.io mirror (legacy)

[`scripts/deploy.R`](scripts/deploy.R) pushes to shinyapps.io (`rsconnect::setAccountInfo(...)`
first). shinyapps.io retires 2026-12-31 — prefer Connect Cloud above.
